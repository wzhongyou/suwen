## 2. 主流产品架构对比

### 2.1 共通的五层架构

所有主流 AI 搜索产品都遵循这五层：

```
用户查询
    │
    ▼
┌──────────────────────────────────┐
│  第一层：查询理解                   │
│  意图分类 / 查询改写 / 查询扩展       │
│  小模型做路由，大模型做复杂改写        │
└──────────────┬───────────────────┘
               ▼
┌──────────────────────────────────┐
│  第二层：混合召回                   │
│  倒排（关键词） + 向量（语义）并发检索  │
│  Perplexity: Vespa               │
│  Kimi: Lindorm 全文+向量          │
│  各产品: 普遍用BM25+Embedding      │
└──────────────┬───────────────────┘
               ▼
┌──────────────────────────────────┐
│  第三层：多阶段排序                 │
│  粗筛 → RRF融合 → Cross-Encoder精排 │
│  融合信号：相关性 + 新鲜度 + 权威性    │
│  Perplexity: 两阶段ML排序          │
└──────────────┬───────────────────┘
               ▼
┌──────────────────────────────────┐
│  第四层：片段抽取 & 上下文工程         │
│  从文档中抽取最相关片段               │
│  细粒度 sub-document spans         │
│  上下文窗口管理（去重、截断、排序）     │
└──────────────┬───────────────────┘
               ▼
┌──────────────────────────────────┐
│  第五层：LLM 生成 + 引用            │
│  约束：只能基于检索到的片段生成        │
│  每个主张附来源引用                  │
│  流式输出（SSE）                    │
└──────────────┬───────────────────┘
               ▼
           搜索呈现
```

| 维度 | Perplexity | 秘塔 | Kimi | You.com | Google |

## 3. 关键技术决策分析

### 3.1 查询理解

**主流做法**：用小模型做意图分类和查询改写，大模型留给最终生成。

```
用户输入: "怎么提高数据库性能"
     │
     ▼
意图分类器 (小模型)
     ├─ 意图: technical_howto
     ├─ 领域: database
     └─ 改写扩展 → ["数据库性能优化方法", "SQL调优最佳实践",
                     "数据库索引优化", "查询性能提升技巧"]
```

**参考实现**:
- ThinkQE: 思考链扩展 + 检索反馈迭代优化
- Query Fan-Out (Google): 单查询扩展为多个合成子查询并发检索
- 秘塔: MoE门控网络做任务分流

**我们的方案**: Go 实现，小模型调用 llmgate 做意图分类+改写，< 200ms

### 3.2 混合召回与融合

**行业标准**: 倒排 + 向量并发 → RRF (k=60) 融合

```
ctx, cancel := context.WithTimeout(ctx, 2*time.Second)
defer cancel()

var wg sync.WaitGroup
wg.Add(2)

go func() { results.bm25 = vortex.Search(ctx, query); wg.Done() }()
go func() { results.vector = proximia.Search(ctx, embedding); wg.Done() }()

wg.Wait()
merged := rrf.Fuse(results.bm25, results.vector, rrf.WithK(60))
```

**关键参数**:
- RRF k=60（社区默认值）
- 每路召回 50-100 条候选
- 向量侧权重可调（探索型查询提高向量权重，导航型查询提高 BM25 权重）

### 3.3 精排策略

| 方案 | 延迟 | 精度 | 成本 | 适用场景 |
|------|------|------|------|---------|
| RRF only | < 10ms | 中等 | 零 | 低延迟要求 |
| + Cross-Encoder | +50-200ms | 高 | 低(GPU) | 生产推荐 |
| + LLM Rerank | +500ms+ | 最高 | 高 | 高价值查询 |

**推荐方案**: RRF 粗排(100条) → Cross-Encoder 精排(30条) → 返回 Top-10

### 3.4 答案生成与引用

**核心原则**（来自 Perplexity 等产品）:
> "没检索到的就不要说"——LLM只能基于检索片段生成，每个主张附引用

```
System Prompt 关键约束:
1. 你只能基于提供的【检索结果】回答
2. 每个事实性陈述必须标注来源编号 [1] [2]
3. 如果检索结果不包含答案，明确说"未找到相关信息"
4. 不要编造检索结果中不存在的数据、日期、数字
```

**我们的差异化**:
- Go 原生，单二进制分发（vs Python生态的部署复杂度）
- 自有倒排引擎(vortex) + 向量引擎(proximia)，不依赖外部搜索API
- 不是 RAG 框架，是完整搜索引擎产品
